# MLightCAD Node spike — report

**Date**: 2026-04-22
**Decision gate**: 🔴 **RED — do NOT pursue Phase 2 with MLightCAD on the backend in its current form.**

---

## TL;DR

MLightCAD is a **browser-targeted** stack. Trying to run its DWG parser
under Node hits two independent, **unrelated** upstream blockers that
cannot be worked around without forking and rebuilding the packages.

| Q | Question | Answer |
|---|---|---|
| Q1 | Pure `import` works under Node? | ✅ Yes — `data-model` (246 exports) + `libredwg-converter` (1 export) load cleanly under Node 25, no `window`/`document` errors |
| Q2 | libredwg WASM loads + parses a real DWG? | ❌ **NO** — blocked by 2 upstream bugs (see below) |
| Q3 | Duration / RSS / output size on real samples? | n/a — Q2 blocked |
| Q4 | Ontology JSON shape? | n/a — Q2 blocked. Shape is documented in `@mlightcad/data-model` source though (`AcDbDatabase`) |
| Q5 | `worker_threads` substitution viable? | ⚠️ Polyfill (`web-worker` npm pkg) loads + spawns worker, but the worker itself dies inside emscripten init (see Bug #2) |

---

## Detailed findings

### ✅ Q1 — pure import (passes)

```text
node -e "import('@mlightcad/data-model').then(m=>console.log('OK',Object.keys(m).length))"
# OK 246

node -e "import('@mlightcad/libredwg-converter').then(m=>console.log('OK',Object.keys(m).join(',')))"
# OK AcDbLibreDwgConverter
```

No browser globals required at module-init time.

### ❌ Q2 — blocking bug #1: converter hard-codes Web Workers

`libredwg-converter/dist/libredwg-converter.js`, class
`AcDbLibreDwgConverter` constructor:

```js
constructor(e = {}) {
  super(e);
  e.useWorker = !0;   // forced — overrides any user value
  e.parserWorkerUrl ||= "/assets/libredwg-parser-worker.js";
}
```

`parse()` then has only one branch:

```js
async parse(e, n) {
  if (t.useWorker && t.parserWorkerUrl) { /* spawn Worker */ }
  else throw new Error("dwg converter can run in web worker only!");
}
```

**No escape hatch.** Even if we polyfill `Worker`, we still must succeed
running the worker bundle.

### ❌ Q2 — blocking bug #2: libredwg-web ESM is broken under Node

`libredwg-web/dist/libredwg-web.js` (the WASM init):

```js
if (ENVIRONMENT_IS_NODE) {
  const { createRequire } = await Promise.resolve()
    .then(() => __viteBrowserExternal);   // ← bug: this is an empty {}
  var require2 = createRequire(import.meta.url);  // TypeError
}
```

`__viteBrowserExternal` is generated by Vite when you tell it
`node:module` is a browser-external. The result baked into the bundle:

```js
const __viteBrowserExternal = Object.freeze(Object.defineProperty({},
  Symbol.toStringTag, { value: "Module" }));   // ← frozen empty object
```

So the Node branch destructures `createRequire` from `{}` and crashes.
**Cannot be monkey-patched at runtime** because the variable is local to
the bundle scope, not a global.

The `.umd.cjs` build exists at `dist/libredwg-web.umd.cjs` but is **not
exposed in `package.json#exports`**, so Node refuses to resolve it
without a fork.

### ⚠️ Q5 — worker polyfill probe (partial)

Tried `web-worker` npm package + `parserWorkerUrl` set to the bundled
file URL. The worker spawns (RSS jumps from 56 → 80 MB, parse latency
went from immediate failure to 124 ms then crash) — but dies inside the
worker with the opaque minified error:

```text
Error: Failed to parse drawing due to error: 'C is not a function'
```

This is almost certainly the same `createRequire` failure as Bug #2,
masked by minification + the worker boundary swallowing the original
stack trace.

## What WOULD it take to make this work?

To unblock spike-style usage, one of:

| Approach | Cost | Risk |
|---|---|---|
| **A. Fork all three packages**: patch `libredwg-converter` to add a non-worker path, patch `libredwg-web` to fix the Vite externalization, expose CJS bundle in exports | 2-5 days + ongoing maintenance every upstream release | 🔴 High — locks us to a fork forever |
| **B. Headless Chromium (Playwright) parser service**: run the existing browser bundle in headless Chrome, expose a Node IPC interface | 2-3 days initial + ~100 MB Chromium per worker, ~500 ms startup per parse | 🟡 Medium — heavy, but stable |
| **C. Patch only the runtime via dynamic rewrite**: read libredwg-web ESM bytes, regex-replace the broken `createRequire` import, eval | 1 day | 🔴 High — fragile, breaks on any minor upstream change |

None of these is "1-2 day spike result enables Phase 2".

## Recommendation

Do **NOT** invest in MLightCAD on the backend. Instead, pick from:

### Option 1 — Keep MLightCAD as a browser-only viewer (Phase 0 reality)

- Status quo of the just-shipped Phase 0 commit.
- Left-side ontology stays sourced from the existing Python DXF parser.
- MLightCAD = visual A/B comparison only (which is what the boss asked
  for in the first place).
- **Cost**: zero additional work.
- **Lost capability**: no second ontology source for LLM A/B.

### Option 2 — Browser → Backend ontology export (deferred)

- When the user opens a Site, the iframe-side MLightCAD parses the DWG
  in the browser as it already does today, then `POST`s the resulting
  `AcDbDatabase` JSON back to a new endpoint
  `POST /dashboard/runs/{id}/semantic`.
- Backend just stores it in `site_model_semantic` jsonb column (as in
  the original Phase 2 plan), no parsing needed server-side.
- **Cost**: ~1 day (new endpoint + alembic migration + iframe ↔ parent
  postMessage bridge).
- **Won**: still get the second ontology source for LLM A/B, with
  almost none of the infrastructure complexity.
- **Lost**: cannot regenerate ontology server-side without the user
  re-opening the page (acceptable for a M2.5 / M3 scope).

### Option 3 — Drop MLightCAD ontology ambition entirely

- Stay with the existing Python DXF baseline + LLM enrichment.
- Skip Phase 2 + Phase 3 of the MLightCAD plan.
- Revisit only if the boss explicitly needs it.

---

## Recommended next step

**Option 1 + queue Option 2 for after current M2.5 work lands.** This
preserves the boss-visible value of Phase 0 (dual rendering A/B) without
locking us into a fork-and-maintain trap, and leaves a clean ~1-day
follow-up to optionally get the second ontology if needed.
