# ProLine CAD 测试用例分析 — 大模型技术必要性论证

**版本**：v2.0（修订：增加反面论证 & 诚实评估）  
**日期**：2026-04-10  
**关联文档**：TEST_CASE_DESIGN.md v1.0、关键技术验证计划 v1.0  
**分析范围**：120 条测试用例（184 个 test method），覆盖 10 个 Spike + 跨 Spike 集成  

---

## 〇、核心问题：为什么要用大模型？

> **质疑（用户/评审方视角）**：  
> "PoC 只有 3 份 SOP + 5 份 FAA PDF + 4 份 DXF。SpaCy + 规则引擎处理 8 份文档完全够用。  
> LLM 带来幻觉风险、API 成本、延迟问题——用在这里是不是在造概念？"

这是一个合理的挑战。本文档的目的是 **直面这个问题**，逐条分析 120 条测试用例中哪些真正需要大模型、哪些是"有 LLM 更好"但传统方式也能过、哪些完全不需要。

---

## 一、诚实分类：三级评估而非二元对立

在 120 条测试用例中，按大模型的 **实际必要性** 重新分为三级：

| 分类 | 用例数 | 占比 | 含义 |
|------|--------|------|------|
| **A — LLM 不可替代** | 12 | 10% | 传统方式在 PoC 数据上也无法达到 Go/No-Go 阈值 |
| **B — LLM 优于传统，但传统也可通过** | 30 | 25% | 传统方式可通过阈值，但开发成本高或扩展性差 |
| **C — 传统方式更优或无需 LLM** | 78 | 65% | 确定性算法/数学/协议，LLM 无用武之地 |

**与 v1.0 的关键差异**：v1.0 将 A+B 合并为"LLM 显著优势 42 条"，掩盖了一个事实——其中 30 条在 PoC 阶段传统方式也能过。

---

## 二、A 类 — LLM 不可替代（12 条用例）

这 12 条有一个共同特征：**任务本身要求的能力超出了规则引擎/正则/SpaCy 的能力边界**，不是"用 LLM 更好"，而是"不用 LLM 做不到"。

### A1：隐式语义矛盾检测（2 条）

**涉及用例**：S5-TC02（SOP_B 矛盾）、S5-TC03（SOP_C 矛盾）

| 矛盾类型 | 示例 | 规则引擎能力 | LLM 能力 |
|----------|------|-------------|---------|
| **显式数值矛盾** | "间距 ≥5m" vs "间距 ≤3m" | ✅ 提取数值+比较区间 | ✅ 同样可行 |
| **隐式流程矛盾** | "先在 180°C 下固化" vs "全程在常温环境下操作" | ❌ 无法理解"180°C 固化"与"常温操作"的逻辑冲突 | ✅ 理解语义蕴含 |
| **跨上下文矛盾** | Ch.3 "铆钉间距不少于 3D" vs Ch.7 "间距按 2.5D 执行" | ❌ 跨章节共指消解超出规则范围 | ✅ 长上下文关联 |

**SOP_B 的 2 对矛盾中 ≥1 对是隐式流程矛盾；SOP_C 的 3 对矛盾中 ≥1 对是跨段落矛盾。**  
规则引擎对数值型矛盾可检出，但测试用例要求"检出 **所有** 矛盾对"（含隐式）——传统方式不可能通过。

**不可替代原因**：矛盾检测的通过条件要求"检出矛盾数 = Gold Standard 矛盾数"。隐式/跨段落矛盾只有语义推理能处理。

---

### A2：跨段落 ProcessGraph DAG 提取（4 条）

**涉及用例**：S5-RT04-01~04

任务：从 FAA AC43.13-1B 章节中提取工艺步骤的有向无环图（DAG）。

| 难点 | 传统方式困难 |
|------|-------------|
| 步骤分布在 Ch.3 不同小节中 | 规则只能处理连续段落，跨小节需人工编写关联规则 |
| 隐式依赖："去毛刺后方可进行铆接" → 去毛刺→铆接 | 需 NLP 识别"后方可"= 时间先后+ 依赖关系 |
| 条件分支："若为盲铆钉，跳过背面目视检查" | 规则引擎需列举所有 if/then 模式 |
| 步骤粒度不一：有的段落 1 步，有的段落含 3 子步 | 步骤边界识别需语义判断 |

**为何 SpaCy 不够**：SpaCy 的依存句法分析器（dependency parser）处理单句效果好，但 **跨段落因果关系抽取** 需要篇章级别推理——这是 2024 年 NLP 学术界仍在研究的开放问题，非 SpaCy 开箱即用。  
**工时对比**：用 SpaCy 实现跨段落 DAG 提取需要自研篇章关系模型（预估 20+ 人天且准确率不确定），LLM prompt 2-3 人天。

### A3：646 页大文档多工艺联合推理（2 条）

**涉及用例**：S5-RT02-04（多块约束去重合并）、S5-RT05-02（跨块一致性）

FAA AC43.13-1B 全书 646 页覆盖 4 种工艺。分块后同一约束可能在不同块中以不同措辞出现：
- Ch.3: "The hole diameter shall be 1/32 inch larger than the rivet shank diameter"
- Ch.7: "Ensure drilled hole provides 0.8mm clearance above fastener diameter"

**去重要求**：识别这两句话描述的是 **同一条约束**（孔径 > 钉径的间隙要求）。

| 方法 | 能否识别为同一约束 |
|------|-------------------|
| 字符串精确匹配 | ❌ 完全不同 |
| Jaccard 相似度 | ❌ 共同词极少 |
| 编辑距离 | ❌ 距离很大 |
| TF-IDF + cosine | ⚠️ 勉强（"diameter" 命中但排序不确定） |
| **Embedding cosine** | ✅ 语义相近（fastener ≈ rivet, clearance ≈ 1/32 inch larger） |

**不可替代原因**：去重的通过条件是"合并后无逻辑重复约束"。跨块跨措辞的语义去重在 8 份文档场景下已出现（646 页足以产生），不是"未来扩展"才遇到的问题。

---

### A4：source_ref 回溯（对改写/概括性引用场景）（2 条）

**涉及用例**：S5-TC04、S5-RT03-03

通过条件：source_ref 准确率 ≥ 90%。

当 LLM 提取的约束描述与原文 **完全一致** 时，字符串匹配/TF-IDF 都能回溯。但实际场景：
- 原文：*"Rivets shall be driven with a rivet gun and bucking bar, maintaining a squeeze force of 15-60 psi"*
- 提取结果：*"铆接压力范围：15-60 psi (使用铆枪+顶铁)"*

这里发生了 **翻译 + 压缩 + 重组**——字符串匹配和 TF-IDF 在此场景覆盖率约 50-65%，达不到 90% 阈值。

**但要诚实**：如果强制 LLM 输出原文原句（不做翻译/压缩），传统 TF-IDF 可达 90%。这取决于约束提取的 prompt 设计——如果 prompt 要求"直接引用原文不做改写"，则传统回溯也能工作。  
**判定**：在允许 LLM 自由生成约束描述的系统设计下不可替代；若约束提取改为"逐句标注"模式，则可降级为 B 类。

### A5：Prompt 策略对比评估（2 条）

**涉及用例**：S5-TC05（零样本/少样本/CoT 对比）、S5-TC07（JSON structured output 校验）

这两条测试用例的测试对象就是 LLM 本身的行为（零样本 vs 少样本谁的 Precision 更高？structured output 是否 100% 可解析？）。**传统方式不存在对应概念**——这不是"哪个更好"的问题，而是"只有 LLM 才有这个测试需求"。

归入 A 类不是因为 LLM "更优"，而是因为这是 LLM 固有的验证需求。

## 三、B 类 — LLM 优于传统，但传统也能通过（30 条）

### 这一类的本质：PoC 阶段传统方式可行，但对项目没有长期价值

以下 30 条用例，如果用传统方式实现，**可以通过 Go/No-Go 阈值**（因为 PoC 数据规模小、文档已知）。但这意味着 PoC 验证的是"规则引擎在固定 8 份文档上能工作"——这不证明任何有技术意义的事情。

### B1：S5 常规约束提取（19 条）

**涉及用例**：S5-TC01, S5-L4, S5-RT01-01~05, S5-RT02-01~03, S5-RT02-05, S5-RT03-01~02, S5-RT03-04, S5-RT05-01, S5-RT05-03, S5-TC06

#### 传统方式可行性分析（对本 PoC 的 8 份文档）

| 文档 | 格式 | 传统提取可行性 | 预估 Precision |
|------|------|---------------|---------------|
| SOP_A (12 条约束) | 我们自己写的 Markdown | ✅ 格式规整，正则可穷举 | ~0.90+ |
| SOP_B (18 条约束) | 我们自己写的 Markdown | ✅ 同上 | ~0.85+ |
| SOP_C (25 条约束) | 我们自己写的 Markdown | ✅ 同上 | ~0.85+ |
| FAA AC43-214 (21pp) | 正式 PDF，格式规范 | ⚠️ 可行但需写专用解析器 | ~0.75+ |
| FAA AC21-26A (15pp) | 正式 PDF，格式规范 | ⚠️ 同上 | ~0.75+ |
| FAA AC145-10 (41pp) | 正式 PDF | ⚠️ 需更多规则 | ~0.70+ |
| FAA AC21.303 (32pp) | 正式 PDF | ⚠️ 需更多规则 | ~0.70+ |
| FAA AC43.13-1B (646pp) | 大文档，多工艺 | ❌ 规模过大，规则开发不现实 | ~0.50-0.60 (仅单章) |

**诚实结论**：  
- 3 份 SOP：传统方式可以达到 Precision ≥0.80。**因为 SOP 是我们自己用 Markdown 写的合成数据**，格式完全可控。  
- 4 份小 FAA PDF（21~41pp）：传统方式也可达标，但每份需单独写解析器（约 2-3 人天/份 = 共 8-12 人天）。  
- FAA AC43.13-1B（646pp）：这是传统方式真正吃力的地方，但这一份文档的提取只涉及 4-5 条用例（S5-RT01-03, RT02-04, RT05-01~03）。

#### 传统方式的真正代价

| 项目 | LLM 方式 | 传统规则方式 |
|------|---------|-------------|
| 3 份 SOP 解析器 | 0 人天（同一 prompt） | 1-2 人天（格式固定，规则简单） |
| 4 份小 FAA PDF 解析器 | 0 人天（同一 prompt） | 8-12 人天（每份需调试规则） |
| 1 份 646pp FAA 大文档 | 0 人天（同一 prompt + 分块） | 10-15 人天（需多章节规则 + 大量调试） |
| PoC 总计 | **3-5 人天**（prompt 设计 + 评估） | **19-29 人天** |
| **生产扩展**：增加 20 份文档 | +0 人天 | **+40-60 人天**（每份新规则） |

**关键论点**：传统方式在 PoC 阶段 **能做**，但产出物是 8 份文档的专用规则——零复用价值。LLM 方式的产出物是 1 套 prompt + 评估管道——直接可用于生产。

#### 此处 LLM 的真实优势不在准确率，在经济性

传统方式可以通过阈值（在已知固定文档上调规则，总能调到 ≥0.80）。LLM 的优势是：
- **开发成本**：5 人天 vs 19-29 人天（3-6x）
- **扩展成本**：追加 0 人天 vs 40-60+ 人天
- **PoC 验证价值**：证明 prompt 泛化能力 >> 证明规则覆盖率

### B2：S1 图层语义映射（5 条）

**涉及用例**：S1-TC02, S1-RT02-01~04

#### 传统方式可行性分析

| 文件 | 层数 | 传统可行性 |
|------|------|-----------|
| wood.dxf | 11 层 | ✅ 手写映射表 10 分钟 |
| fish.dxf | 118 层 | ⚠️ 手写映射表 2-3 小时 |
| cold_rolled.dxf | 5 层（含俄语） | ⚠️ 3 个英文 + 1 个俄语 + Defpoints，查翻译后手写规则 30 分钟 |
| wood_factory.dxf | 9 层 | ✅ 手写映射表 20 分钟 |

**4 份文件总层数**：~143 个（去重后更少）。手动建映射表 **总计半天工作量**，可达到 ≥85% 分类率。

**诚实结论**：在 PoC 这 4 份已知 DXF 上，LLM 没有显著优势。传统查表法完全够用且更快（规则写完后执行 0ms vs LLM 调用 500ms+）。

LLM 的优势仅在 **生产阶段面对未知底图** 时才体现。但 PoC 测试的 4 份文件是固定的——事先就知道所有层名。

### B3：S9 RAG 检索（6 条）

**涉及用例**：S9-TC01~TC04, S9-L4, S9-PERF-STAT  
**不含 S9-TC05 和 S9-TC06**（这两条专门评估混合策略和 Golden 集，见下文单独分析）

#### 传统方式可行性分析

**语料库规模**：~8 份文档（3 SOP + 5 FAA），分块后预计 50-100 个 chunk。

| 维度 | BM25 + 同义词扩展 | Embedding + HNSW |
|------|-------------------|-----------------|
| 精确查询 Recall@5 | **0.90+**（关键词完全命中） | 0.85+（向量近似） |
| 同义词查询 Recall@5 | 0.70+（需手工同义词表） | **0.85+**（天然语义） |
| 索引成本 | 0（BM25 纯文本倒排索引） | 需 GPU 生成 embedding |
| 查询延迟 | **<10ms** | 50-200ms |
| PoC 语料下总 Recall@5 | 预估 **0.80+** | 预估 0.85+ |

**诚实结论**：50-100 个 chunk 的小语料库上，BM25 + 手工同义词表（~20 条同义词，2 小时工作量）有较大概率达到 Recall@5 ≥0.80。**Embedding 的优势被小语料库稀释**——BM25 的稀疏命中在小库上已足够精确。

LLM/Embedding 的真正优势在 **千级以上 chunk** 的语料库（同义词表维护成本爆炸、长尾查询增多）。

## 四、C 类 — 传统方式更优（78 条用例）

这 78 条涉及确定性计算、几何运算、数学仿真、协议验证、渲染输出。**不存在"用 LLM 更好"的可能性**，此处不再详细论证（v1.0 已充分分析），仅列汇总表：

| Spike | 用例数 | 核心技术 | 为何传统更优 |
|-------|--------|---------|-------------|
| S1 实体解析/性能/坐标 | 28 | ezdxf + numpy | DXF 是结构化二进制，确定性解析 0 误差 |
| S2 MCP 协议 | 1 | MCP SDK | 协议字段二值断言 |
| S3 碰撞检测 | 9 | R-Tree + Shapely | 实时几何运算 ≤20ms，精确到 mm² |
| S4 DES 仿真 | 7 | SimPy | 数学仿真，固定 seed 结果 identical |
| S6 Temporal 编排 | 8 | Temporal SDK | 工作流状态机，确定性状态转移 |
| S7 3D 场景数据 | 5 | Three.js JSON | 结构化变换（位置/旋转/缩放） |
| S8 PINN | 6 | PyTorch | 物理信息深度学习（非 LLM） |
| S10 报告渲染 | 8 | Jinja2 + WeasyPrint | 模板确定性渲染 |
| INT 集成（非 S5 相关） | 6 | 跨模块 | 数据流断言 |

> **判断铁律**：输入是结构化格式 + 操作是确定性计算 + 输出要求精确匹配 → 传统方式绝对优于 LLM。

## 五、预设反驳 — 回应"为什么不用传统方式"

### 反驳 1："8 份文档写规则就够了，不需要 LLM"

**承认**：是的，3 份 SOP（我们自己写的 Markdown）用正则匹配就够了。4 份短 FAA PDF 写专用解析器也可行。

**但对方忽略了**：
1. **PoC 的目的不是"跑通 8 份文档"，而是验证技术路线的生产可行性**。用规则跑通 8 份文档证明了什么？证明了"规则在已知文档上有效"——这是同义反复，毫无技术验证价值。
2. **投入产出失衡**：规则方案 19-29 人天开发的产出物在生产阶段全部作废（新文档需新规则）。LLM 方案 3-5 人天的 prompt + 评估管道直接沿用到生产。
3. **646 页文档不可忽视**：FAA AC43.13-1B 是真实存在的大规模文档。即使 PoC 只从中提取 1-2 个章节，也已经超出了"写规则就够"的范围。

### 反驳 2："LLM 有幻觉，规则引擎不会凭空捏造"

**承认**：这是 LLM 的 **真实劣势**。规则引擎的 hallucination rate = 0%，LLM 的指标是 ≤10%。

**但对方忽略了**：
1. 规则引擎的 hallucination = 0% 但 **recall 很低**（遗漏的约束不算幻觉，但同样危险——在航空制造场景，漏检约束可能比多检更严重）
2. PoC 的 S5-TC06 专门验证幻觉控制——**这正是用 PoC 来验证 LLM 能否将幻觉率压到可接受范围**的需要
3. 缓解机制已设计：temperature=0.1 + source_ref 强制回溯 + structured output

### 反驳 3："Embedding 检索在 8 份文档上没优势，BM25 就够了"

**承认**：在 50-100 个 chunk 上 BM25 确实能达标。

**但 PoC 的 Spike-9 不是在验证"8 份文档能否检索"**：
1. S9 验证的是 **Embedding 索引 + 向量检索管道** 是否工作（端到端通路）
2. 即使在小语料上 BM25 可达标，PoC 仍需验证 Embedding 路径——因为生产环境是 Embedding 路径
3. **混合检索框架** (S9-TC05) 本身包含 BM25——不是"要么 BM25 要么 Embedding"，而是两者协同

### 反驳 4："图层映射手写查表半天搞定"

**完全承认**：在 PoC 这 4 份 DXF 上，**查表比 LLM 更合适**。

建议将 S1-TC02 / S1-RT02-01~04 的实现改为 **双通道对比验证**：
1. 第一通道：规则查表（PoC 基线，预期 ≥85%）
2. 第二通道：LLM 语义分类（对比实验，预期 ≥85%）
3. 对比输出：记录两种方式在每份 DXF 上的分类差异 → 输出为 Spike-1 验证报告的一部分

这样 PoC 不仅测试了"能否达标"，还量化了"LLM vs 查表在相同数据上的差距"——为生产阶段的技术选型提供数据依据。

## 六、修订后的决策矩阵

### 6.1 PoC 阶段技术选择总览（按真实必要性）

| Spike | 核心技术 | LLM 必要性 | 说明 |
|-------|---------|-----------|------|
| **S1** DWG 解析 | ezdxf (传统 parser) | 🟡 B 类：图层映射可双通道对比 | 查表能过，但不证明什么 |

### 6.2 大模型使用的真实覆盖边界

```
120 条测试用例
├── C 类：78 条 (65%) — 传统方式更优，不需要讨论
├── B 类：30 条 (25%) — 传统可通过 PoC 阈值，但不具备生产价值
```

## 七、风险清单（传统方式同样有风险）

| 风险 | LLM 方式 | 传统方式 |
|------|---------|---------|
| **幻觉/错误输出** | ≤10% hallucination | 0% hallucination 但 recall 低（遗漏约束） |
| **不确定性** | 同输入不同输出（缓解：temperature=0.1） | 确定性输出 |
| **延迟** | 500ms-30s/call | <10ms |
| **PoC→生产扩展成本** | **~0 人天**（同 prompt） | **40-60+ 人天**（新文档新规则） |
| **维护成本** | 模型升级可能影响 prompt | 规则随文档格式变化全部重写 |
| **API 成本** | ~$0.01-0.10/call | 0 |
| **技术债** | prompt 工程经验可复用 | **8 份文档专用规则 = 纯技术债** |
| **PoC 验证价值** | 证明 LLM 在航空文档上的提取能力 | **证明规则在已知文档上有效（同义反复）** |

## 八、结论与建议

### 如果被问"120 条用例中 LLM 真正不可替代的有多少？"

**诚实回答：12 条（10%）**。但这 12 条覆盖的是系统最难的技术点（隐式矛盾、跨段落 DAG、语义去重），恰恰是 PoC 需要验证的核心技术风险。

### 如果被问"那其他 30 条 B 类为什么不用传统方式？"

**回答**：可以。但 PoC 的目的是 **验证技术路线**，不是跑通一套"已知答案的确定性流程"。用规则引擎处理已知的 8 份文档，通过了 Go/No-Go 阈值，只能证明"规则在这 8 份上有效"——这不需要做 PoC 来验证。

**LLM 方案即使在 PoC 小规模数据上只是"和传统方式差不多"，它验证的东西完全不同**：
1. prompt 对新文档类型的零样本泛化能力
2. 结构化输出（JSON ConstraintSet）的可靠性
3. 幻觉率在受控条件下是否可接受
4. 大文档分块→提取→合并的端到端管道

这些才是 PoC 的意义。

### 建议调整：引入双通道对比实验

对 B 类 30 条用例，建议在 Spike-1 和 Spike-5 中增设 **传统方式基线对比**：

| Spike | 双通道对比内容 | 期望输出 |
|-------|---------------|---------|
| S1 图层映射 | 查表 vs LLM，每份 DXF 对比分类率 + 耗时 | 量化"LLM 在已知数据上的边际收益" |
| S5 约束提取 | SpaCy+规则 vs LLM，对 SOP_A/B/C 对比 Precision/Recall | 量化"同一评估标准下两种方案的差距" |
| S9 RAG | BM25 vs Embedding vs Hybrid，同一 query 集对比 Recall@5 | 量化"小语料库下 Embedding 的边际收益" |

这样即使 LLM 在 PoC 上只是"略好一点"，也有数据证明差距幅度——比空讲"LLM 有优势"有说服力。
